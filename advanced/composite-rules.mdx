<Warning>
Composite rules are an experimental feature introduced in v8.28.0. The behavior and configuration format are subject to change in future versions.
</Warning>

A composite rule is a primary rule that only fires when one or more auxiliary rules also match within the same fragment of content. This lets you require multiple signals to co-occur before reporting a finding, reducing false positives for rules that are ambiguous in isolation.

## Concept

A standard rule fires whenever its `regex` matches. A composite rule adds a dependency: the primary rule's regex must match **and** each `required` auxiliary rule must also match. The proximity constraints (`withinLines`, `withinColumns`) control how close the required matches must be to the primary match.

A **fragment** is the unit of content Gitleaks processes at once вҖ” typically a file, a section of a file, or a git diff hunk.

## Configuration

Add a `[[rules.required]]` table to the primary rule. Each `[[rules.required]]` entry requires:

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | Yes | The `id` of the auxiliary rule that must also match |
| `withinLines` | int | No | Primary and auxiliary matches must be within this many lines of each other |
| `withinColumns` | int | No | Primary and auxiliary matches must be within this many columns of each other |

```toml
[[rules]]
id = "my-primary-rule"
description = "Detects a secret only when a known hostname is nearby"
regex = '''(?i)password\s*=\s*["']?([a-z0-9]{16,})'''
secretGroup = 1
keywords = ["password"]

  [[rules.required]]
  id = "my-hostname-rule"
  withinLines = 5

[[rules]]
id = "my-hostname-rule"
description = "Matches a known internal hostname"
regex = '''internal\.example\.com'''
skipReport = true
```

### `skipReport`

Set `skipReport = true` on any rule that should only be used as an auxiliary rule and never generate its own findings. This is the typical pattern for helper rules used exclusively in `[[rules.required]]` references.

## Proximity modes

The `withinLines` and `withinColumns` fields define four distinct matching behaviors.

## Full example

This example detects an AWS secret access key only when the corresponding access key ID is found nearby, preventing standalone secret-shaped strings from triggering a finding.

```toml
[[rules]]
id = "aws-secret-key-with-id"
description = "AWS Secret Access Key found near an Access Key ID"
regex = '''(?i)aws.{0,20}['"]?([a-zA-Z0-9\/+]{40})['"]?'''
secretGroup = 1
entropy = 3.5
keywords = ["aws"]

  [[rules.required]]
  id = "aws-access-key-id"
  withinLines = 5

[[rules]]
id = "aws-access-key-id"
description = "AWS Access Key ID"
regex = '''(AKIA[0-9A-Z]{16})'''
secretGroup = 1
skipReport = true
```

## Scan type considerations

Composite rules are more effective in `dir` scans than in `git` scans. In `git` mode, Gitleaks only examines additions in the diff, so if the primary and required matches are not both part of the same diff hunk, they may not appear in the same fragment and the required constraint will not be satisfied.
