---
title: 'Allowlists'
description: 'Suppress false positives using rule-level and global allowlists, inline comments, and the .gitleaksignore file.'
---

Allowlists tell gitleaks to ignore findings that match certain criteria. There are two scopes: **rule-level** allowlists that apply only to a specific rule, and **global** allowlists that apply across all rules. Within each scope you can match on commit SHAs, file paths, regex patterns, or stop words.

## Rule-level allowlists

Add `[[rules.allowlists]]` tables inside a `[[rules]]` block to suppress findings from that rule only:

```toml
[[rules]]
id    = "my-api-key"
regex = '''myservice-[0-9a-zA-Z]{32}'''

    [[rules.allowlists]]
    description = "Ignore test fixtures and go module files"
    condition   = "OR"
    commits     = ["commit-A", "commit-B"]
    paths       = [
      '''go\.mod''',
      '''go\.sum''',
    ]
    stopwords   = ["client", "endpoint"]
```

A finding is ignored if **any** `[[rules.allowlists]]` entry matches (the entries themselves use an OR relationship by default).

<Warning>
  `[rules.allowlist]` (singular, bracket notation) was deprecated in **v8.21.0** and replaced by `[[rules.allowlists]]`. Instances of the singular form still work for backwards compatibility but will be removed in v9. The two forms cannot be mixed in the same rule.
</Warning>

## Global allowlists

Add `[[allowlists]]` tables at the top level (outside any `[[rules]]` block) to apply them across every rule:

```toml
[[allowlists]]
description = "Global allow list"
commits     = ["commit-A", "commit-B", "commit-C"]
paths       = [
  '''gitleaks\.toml''',
  '''(.*?)(jpg|gif|doc)''',
]
regexTarget = "match"
regexes     = [
  '''219-09-9999''',
  '''078-05-1120''',
  '''(9[0-9]{2}|666)-\d{2}-\d{4}''',
]
stopwords   = ["client", "endpoint"]
```

Global allowlists have **higher precedence** than rule-level allowlists. If a global allowlist matches, gitleaks suppresses the finding before evaluating rule-level allowlists.

<Warning>
  `[allowlist]` (singular, bracket notation) was deprecated in **v8.25.0** and replaced by `[[allowlists]]`. The singular form still works for backwards compatibility but will be removed in v9.
</Warning>

## Allowlist fields

<AccordionGroup>
  <Accordion title="description">
    A short human-readable label for the allowlist entry. Appears in debug output.

    ```toml
    description = "Ignore commit A"
    ```
  </Accordion>

  <Accordion title="condition">
    Controls how multiple criteria within the same allowlist entry are combined. Defaults to `"OR"`.

    | Value | Behaviour |
    |-------|-----------|
    | `"OR"` (default) | A finding is suppressed if **any** criterion matches. |
    | `"AND"` | A finding is suppressed only if **all** criteria match simultaneously. |

    ```toml
    condition = "AND"
    ```
  </Accordion>

  <Accordion title="commits">
    An array of commit SHAs. Any finding originating from one of these commits is suppressed. Comparison is case-insensitive.

    ```toml
    commits = ["a1b2c3d4e5f6", "deadbeefcafe"]
    ```
  </Accordion>

  <Accordion title="paths">
    An array of Go regular expressions matched against the file path of the finding. If any path pattern matches, the criterion is satisfied.

    ```toml
    paths = [
      '''go\.mod''',
      '''go\.sum''',
      '''tests/fixtures/.*''',
    ]
    ```
  </Accordion>

  <Accordion title="regexes">
    An array of Go regular expressions matched against the finding value specified by `regexTarget`. If any regex matches, the criterion is satisfied.

    ```toml
    regexes = ['''(?i)example''', '''placeholder-\d+''']
    ```
  </Accordion>

  <Accordion title="regexTarget">
    Selects what the `regexes` patterns are matched against.

    | Value | Matched against |
    |-------|-----------------|
    | `"secret"` (default) | The extracted secret value (the `secretGroup` capture, or full match). |
    | `"match"` | The full regex match string from the rule. |
    | `"line"` | The entire line of content containing the match. |

    ```toml
    regexTarget = "match"
    regexes     = ['''(?i)parseur[il]''']
    ```

    <Note>
      Specifying `regexTarget = "secret"` is equivalent to omitting the field entirely.
    </Note>
  </Accordion>

  <Accordion title="stopwords">
    An array of strings matched against the **extracted secret** value using an Aho-Corasick trie (case-insensitive substring match). Unlike `regexes`, stopwords always target the secret — they are not affected by `regexTarget`.

    ```toml
    stopwords = ["client", "endpoint", "example"]
    ```
  </Accordion>

  <Accordion title="targetRules (global allowlists only, v8.25.0+)">
    Assigns a global `[[allowlists]]` entry to specific rules by ID instead of applying it everywhere. Useful when a common false-positive pattern only occurs in the context of certain rules.

    ```toml
    [[allowlists]]
    targetRules = ["awesome-rule-1", "awesome-rule-2"]
    description = "Test assets trigger false positives in these two rules."
    paths       = ['''tests/expected/._\.json$''']
    ```

    An allowlist with `targetRules` is **not** added to the global list; it is attached only to the named rules.
  </Accordion>
</AccordionGroup>

## AND vs OR condition examples

<Tabs>
  <Tab title="OR condition (default)">
    A finding is suppressed if the commit matches **or** the path matches **or** the stopword is found.

    ```toml
    [[rules.allowlists]]
    description = "Suppress if any condition is met"
    condition   = "OR"
    commits     = ["commit-A", "commit-B"]
    paths       = [
      '''go\.mod''',
      '''go\.sum''',
    ]
    stopwords   = ["client", "endpoint"]
    ```
  </Tab>
  <Tab title="AND condition">
    A finding is suppressed only when **both** the regex matches **and** the path matches.

    ```toml
    [[rules.allowlists]]
    description = "Only suppress parseur matches inside package-lock.json"
    condition   = "AND"
    regexTarget = "match"
    regexes     = ['''(?i)parseur[il]''']
    paths       = ['''package-lock\.json''']
    ```
  </Tab>
</Tabs>

## Inline suppression with gitleaks:allow

For one-off cases, add a `gitleaks:allow` comment on the line containing the secret. Gitleaks will skip that line entirely during scanning.

```python
class CustomClass:
    discord_client_secret = '8dyfuiRyq=vVc3RRr_edRk-fK__JItpZ'  #gitleaks:allow
```

This works in any language. The comment must appear on the same line as the secret.

<Note>
  You can disable recognition of `gitleaks:allow` comments with the `--ignore-gitleaks-allow` flag.
</Note>

## Suppressing findings by fingerprint (.gitleaksignore)

Every finding includes a `Fingerprint` that uniquely identifies it by commit, file, rule ID, and line number:

```
cd5226711335c68be1e720b318b7bc3135a30eb2:cmd/generate/config/rules/sidekiq.go:sidekiq-secret:23
```

Add fingerprints — one per line — to a `.gitleaksignore` file at the root of the repository to permanently suppress those findings:

```
cd5226711335c68be1e720b318b7bc3135a30eb2:cmd/generate/config/rules/sidekiq.go:sidekiq-secret:23
a0b1c2d3e4f5a0b1c2d3e4f5a0b1c2d3e4f5a0b1:src/config.go:generic-api-key:42
```

Use the `-i` / `--gitleaks-ignore-path` flag to specify a different location for the `.gitleaksignore` file.

<Tip>
  The fingerprint approach is ideal for false positives that are genuinely difficult to express as regex or stopword patterns — for example, real-looking random strings in test data or known-safe tokens in documentation.
</Tip>
