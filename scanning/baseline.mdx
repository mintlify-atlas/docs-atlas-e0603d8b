---
title: "Using baselines"
description: "Suppress known findings with a baseline report so gitleaks only surfaces new secrets."
---

A **baseline** is a previously generated gitleaks report (in JSON format) that
represents the known state of secrets in a repository. When you run gitleaks
with `--baseline-path`, any finding that matches an entry in the baseline is
suppressed — only genuinely new findings appear in the output.

This is particularly useful when you first introduce gitleaks to an existing
repository with a long history: you can acknowledge the current state as a
baseline and then use gitleaks to catch only new secrets going forward.

## When to use a baseline

<CardGroup cols={2}>
  <Card title="Large or legacy repositories" icon="building">
    Repositories with years of history may contain many pre-existing findings.
    A baseline lets you suppress those and focus on new commits.
  </Card>
  <Card title="Onboarding gitleaks to CI" icon="circle-check">
    When enabling gitleaks in CI for the first time, a baseline prevents the
    pipeline from immediately failing on findings that were already known and
    accepted.
  </Card>
  <Card title="Incremental scanning" icon="chart-line">
    Run a full scan once to establish a baseline, then use the baseline in
    subsequent scans to catch only secrets introduced after that point.
  </Card>
  <Card title="Suppressing accepted false positives" icon="filter">
    If certain findings are confirmed non-secrets and cannot be fixed, a
    baseline provides a lightweight way to suppress them without modifying
    allowlist configuration.
  </Card>
</CardGroup>

## How it works

When `--baseline-path` is set, gitleaks loads the baseline JSON file and parses
it as a list of previous findings. For every finding detected in the current
scan, gitleaks checks whether an identical finding exists in the baseline by
comparing the following fields:

- `RuleID`
- `Description`
- `StartLine` and `EndLine`
- `StartColumn` and `EndColumn`
- `Match` and `Secret` (skipped when `--redact` is active, since values are masked)
- `File`
- `Commit`
- `Author`
- `Email`
- `Date`
- `Message`
- `Entropy`

The `Fingerprint` field is intentionally excluded from this comparison so that
baseline suppression continues to work correctly even if the fingerprint format
changes between gitleaks versions.

If all compared fields match a baseline entry, the finding is considered **not
new** and is omitted from the report.

## Step-by-step walkthrough

<Steps>
  <Step title="Run an initial scan to create the baseline">
    Scan your repository and save the output as a JSON report. This becomes
    your baseline.

    ```bash
    gitleaks git --report-path gitleaks-report.json .
    ```

    If gitleaks finds secrets, it exits with code `1`. The report is still
    written regardless of exit code.
  </Step>

  <Step title="Review and commit the baseline">
    Inspect the generated `gitleaks-report.json` to confirm you understand
    every finding it contains. Then commit the file to your repository or store
    it in a known location accessible to your CI environment.

    ```bash
    git add gitleaks-report.json
    git commit -m "chore: add gitleaks baseline"
    ```
  </Step>

  <Step title="Run subsequent scans with the baseline">
    Pass the baseline file via `--baseline-path`. The report written to
    `--report-path` will contain only findings that are **not** present in the
    baseline.

    ```bash
    gitleaks git \
      --baseline-path gitleaks-report.json \
      --report-path findings.json \
      .
    ```

    If no new secrets are found, gitleaks exits with code `0`.
  </Step>

  <Step title="Update the baseline when needed">
    After remediating findings or accepting new known secrets, regenerate the
    baseline by running the initial scan command again with a fresh
    `--report-path`. Replace the old baseline file with the new one.

    ```bash
    gitleaks git --report-path gitleaks-report.json .
    ```
  </Step>
</Steps>

## CI example

The following shows how to use a baseline in a CI pipeline where the baseline
file is committed to the repository.

```bash
# Step 1 (run once, locally): create the initial baseline
gitleaks git --report-path .gitleaks-baseline.json .

# Step 2: commit the baseline
git add .gitleaks-baseline.json && git commit -m "chore: gitleaks baseline"

# Step 3 (CI): scan against the baseline on every push
gitleaks git \
  --baseline-path .gitleaks-baseline.json \
  --report-path new-findings.json \
  .
```

## Important notes

<Warning>
  The baseline must be a JSON report generated by gitleaks. Passing a file in
  any other format (CSV, SARIF, etc.) will cause an error:
  `the format of the file ... is not supported`.
</Warning>

<Warning>
  The `--baseline-path` value must be a valid, readable file path. If the file
  cannot be opened or parsed, gitleaks logs an error and continues the scan
  **without** any baseline suppression applied.
</Warning>

<Note>
  The baseline path is resolved relative to the scan source directory. This
  means a baseline file stored at the root of the repository being scanned can
  be referenced with just its filename when the repo root is the scan target.
</Note>

## Flag reference

<ParamField path="-b, --baseline-path" type="string" default="">
  Path to a gitleaks JSON report to use as a baseline. Findings that match
  entries in this file are excluded from the current scan's output.
</ParamField>

<ParamField path="-r, --report-path" type="string" default="">
  Path to write the current scan's findings. When used together with
  `--baseline-path`, only new (non-baseline) findings are written here.
</ParamField>

<ParamField path="-f, --report-format" type="string" default="">
  Output format for `--report-path`. Accepted values: `json`, `csv`, `junit`,
  `sarif`, `template`. The format is inferred from the file extension if not
  set.
</ParamField>
